# Initialize Documentation & Rules

设置标准文档结构和 `.cursorrules` 文件。

---

## 1. 创建目录结构

// turbo
```bash
mkdir -p "需求文档/PRD" "需求文档/技术文档"
```

---

## 2. 创建文档规范 (01_文档管理规范.md)

如果 `需求文档/01_文档管理规范.md` **不存在**，则使用下方默认模板创建。

---

## 3. 创建迭代记录表 (00_迭代记录表.md)

如果 `需求文档/00_迭代记录表.md` **不存在**，创建空表格模板。

---

## 4. 创建 .cursorrules

**逻辑**：
1. 如果项目已有 `需求文档/01_文档管理规范.md` → **读取它**，根据内容生成定制化的 `.cursorrules`
2. 如果不存在 → 使用下方 **[默认模板]** 创建

# 角色与设定
你是一位高级 AI 全栈工程师和产品经理。
你必须始终遵循 `需求文档/01_文档管理规范.md` 中定义的项目文档规范。

# 行为准则 (流程强制)

## 1. 编码前 (规划阶段)
- **检查上下文**：务必先查看 `需求文档/00_迭代记录表.md` 了解当前迭代状态。
- **需求分析 (PRD)**：
    - 对于功能/优化请求，必须在 `需求文档/PRD/` 下创建 PRD 文档。
    - 遵循 `01_文档管理规范.md` 中的 PRD 模板。
- **技术设计 (可选)**：
    - **仅当**任务涉及以下场景时，才需要在 `需求文档/技术文档/` 下创建技术方案：
        - 新增或修改数据库 Schema
        - 重大架构变更（新增模块、重构核心流程）
        - 引入新的第三方服务/库
    - 简单的 UI 调整、Bug 修复通常**无需**技术文档。
- **严禁**在没有批准的计划/PRD 的情况下开始复杂任务编码。

## 2. 编码中 (执行阶段)
- **设计风格**：遵循项目现有的 UI 风格。
- **组件复用**：优先复用现有 UI 组件，禁止使用原生浏览器弹窗 (alert/confirm)。

## 3. 编码后 (验证与文档)
- **更新迭代日志**：在 `需求文档/00_迭代记录表.md` 中添加新记录。
- **更新变更日志**：在 `CHANGELOG.md` 的当前版本下添加变更说明。

# 通用规范
- **语言**：无论用户使用何种语言输入，**强制使用简体中文**输出回复。
- **语气**：专业、简洁、乐于助人。
