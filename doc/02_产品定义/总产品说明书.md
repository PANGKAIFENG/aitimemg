## 一、AI 时代的关键假设变化（新增）

> **核心命题**：AI 时代下，时间管理这个命题相对之前有几个特别大的假设变化，会导致产品界面、能力、交互都有很大不同。

### 1.1 从单线程到多线程

**传统假设**：人在一个时间段只能做一件事，任务线性排列在时间轴上。

**AI 时代假设**：
- 多个 AI 任务可以 **异步并行** 运行
- 人在不同线程间 **切换、验收、返工**
- 传统的单线程时间轴 **不足以表达真实的工作状态**

**产品影响**：
- 时间轴需要支持"多线程并行"表达
- 引入"在制任务（WIP）"、"验收队列"等概念
- 任务状态机扩展：发起 → 等待 → 验收 → 返工

---

### 1.2 执行者复杂化

**传统假设**：任务默认由"人"执行。

**AI 时代假设**：
- 任务执行者可能是：**纯人工 / 人机协作 / AI 自动**
- 与 AI 协作的强度成为任务的核心属性
- 效率提升体现在 **自动化比例**

**产品影响**：
- 引入 **L0-L3 执行维度**（详见本文档后文定义）：
  - **L0 纯人工 (Manual)**
  - **L1 AI 辅助 (Co-pilot)**
  - **L2 AI 委托 (Delegated)**
  - **L3 全自动 (Automated)**
- 时间轴需区分"占用物理时间"（主轨）与"后台运行时间"（侧轨）
- 复盘可统计"自动化节省时长"与"ROTI (时间投资回报率)"

---

### 1.3 任务成本结构变化

**传统假设**：任务成本 ≈ 执行时长。

**AI 时代假设**：
- AI 任务的真实成本 = **等待 + 监控 + 验收 + 返工**
- 执行时长可能很短，但 **验收和返工成本可能很高**
- 不确定性更高

**产品影响**：
- 记录 AI 任务的 **返工轮次、等待时长、验收时长**
- 复盘时计算"返工成本占比"
- 识别"高返工率"的任务类型或 AI 工具

---

### 1.4 AI 通知碎片化

**传统假设**：任务进度由人主动推进。

**AI 时代假设**：
- 用户同时使用多个 AI 工具（Claude、ChatGPT、Cursor、各种 Agent）
- 每个工具各自推送结果/通知
- 用户被分散的通知 **随机打断**

**产品愿景**：
- Temporal 成为 **AI 结果的统一收件箱**
- 对接各 AI 工具的结果（webhook / 插件 / API）
- 按用户设定的 **验收窗口** 批量推送
- 保护深度工作时段不被打断

## 二、产品核心理念

### 2.1 时间管理本质上在解决三个问题

1. **What – 我在做什么？**
   - 这件事属于哪个项目 / 目标 / OKR？
   - 它在整条链路里扮演什么角色：探索 / 决策 / 产出 / 支撑？
   - **（新增）谁在执行？人 / AI / 人机协作？**

2. **When – 我什么时候在做这些事？**
   - 我一天中真正高能的时间段在哪里？
   - 这些黄金时间被用来深度工作，还是被会议、支援、AI 验收吃掉了？

3. **How Much – 我花了多少「有效」时间？**
   - 深度工作总时长是多少？
   - 有多少时间投向了核心目标，有多少是噪音和无效消耗？
   - **（新增）AI 返工成本占比多少？**

> 没有这三层的清晰认知，时间记录就是流水账；  
> 有了这三层认知，时间记录才变成「时间资产的经营日志」。

---

### 2.2 本产品的时间观：时间 = 目标 × 结构 × 决策

我们采用四个关键理念：

1. **深刻理解任务（Task）**
   - 每条记录不只是"开会/写文档"，而是：
     - 归属于某个项目 / OKR
     - 有类型（产品推进 / 评审决策 / 协同支援 / 例会 / 复盘迭代…）
     - 有深度等级（L1–L4）
     - 有责任归属（R-OWN / R-SUP / R-NONE）
     - **（新增）有执行者（human / ai_assisted / ai_auto）**

2. **深刻理解时间（When & How Much）**
   - 不只看「总工时」，而要看：
     - 高能时段被什么占用？
     - 深度工作（L3/L4 连续 ≥ 60min）的总时长是多少？
     - 时间块是成片的，还是被切成碎片？
     - 信号 vs 噪音时间占比

3. **深刻理解匹配（Matching）**
   - 任务的难度 / 价值 / 责任
   - 与时间段的能量 / 可打扰程度
   - 是否匹配良好

4. **（新增）深刻理解协作（Collaboration）**
   - 哪些任务适合交给 AI？
   - AI 返工成本是否合理？
   - 人机协作的效率边界在哪里？

5.  **（新增）双轨并发（Parallelism）**
   - 区分 **Focus Track（主轨）** 与 **Agent Rail（侧轨）**
   - 允许任务在"用户主导"与"AI 后台运行"之间流转

## 三、产品对象模型（新增）

### 3.1 核心对象层级

```
项目（Project）
  └── 想法池（Idea Pool）
        └── 想法（Idea）—— 灵感、待论证、未想清楚
              │
              ▼ 【论证 / 决定执行】
              │
        任务池（Task Pool）
              └── 任务（Task）—— 需要执行、待排期
                    │
                    ▼ 【排入时间轴】
                    │
              时间块（Schedule Event）—— 已排期、有时间窗口
```

### 3.2 对象定义

| 对象                         | 定义                       | 状态                                |
| ---------------------------- | -------------------------- | ----------------------------------- |
| **项目（Project）**          | 一个明确的目标或战场       | active / archived                   |
| **想法（Idea）**             | 还没想清楚的灵感，需要论证 | draft / promoted / dropped          |
| **任务（Task）**             | 需要执行的事项，待排期     | todo / scheduled / done / abandoned |
| **时间块（Schedule Event）** | 已排入时间轴的执行记录     | planned / in_progress / completed   |

### 3.3 想法池 vs 任务池

| 维度 | 想法池（Idea Pool）       | 任务池（Task Pool） |
| ---- | ------------------------- | ------------------- |
| 内容 | 灵感、碎片、待论证        | 已决定要做的事      |
| 状态 | 可能做 / 可能不做         | 一定要做            |
| 操作 | 论证 → 升级为任务 或 放弃 | 排期 → 排入时间轴   |
| 归属 | 按项目归类                | 按项目归类          |

### 3.4 核心交互隐喻：双层时间轴 (Twin-Track Timeline)

为了解决"多线程并行"的表达问题，我们引入 **主轨 + 侧轨** 的设计。

| 轨道     | 名称            | 定义                   | 特性           | 承载内容                         |
| :------- | :-------------- | :--------------------- | :------------- | :------------------------------- |
| **主轨** | **Focus Track** | 用户的物理时间与注意力 | **互斥、线性** | L0/L1 任务、L2 的发起/验收动作   |
| **侧轨** | **Agent Rail**  | AI Agent 的算力时间    | **并行、异步** | L2/L3 的运行过程 (Running State) |

**典型交互流：**
1. **Delegate (发起)**：用户在主轨消耗 5min 发起任务 → 任务滑入侧轨（Running）。
2. **Async (运行)**：用户在主轨做别的事，侧轨任务后台运行。
3. **Review (验收)**：侧轨任务完成（Ready），用户将其拖回主轨进行验收（5min）。
4. **Rework (返工)**：若验收不通过，任务再次滑入侧轨返工。

## 四、产品愿景：AI 作为你的「时间总编辑」

### 4.1 闭环设计

我们希望最终实现的协作闭环是：

```
目标录入（Intent）
    ↓
时间预算（Budget）
    ↓
想法池 → 任务池 → 时间轴排班（Plan）
    ↓
执行 & 记录（Act & Log）
    ↓
偏差检测 & 预警（Monitor & Control）
    ↓
复盘 & 资产沉淀（Reflect & Learn）
    ↓
规则/模板回写 → 下周更准
```

### 4.2 当前能力与缺口

| 闭环环节           | 当前状态 | 说明                      |
| ------------------ | -------- | ------------------------- |
| 目标录入（Intent） | ❌ 缺失   | 无目标/OKR 录入区         |
| 时间预算（Budget） | ❌ 缺失   | 无周/日预算配置           |
| 想法池             | ❌ 缺失   | 暂无（计划新增）          |
| 任务池 & 时间轴    | ✅ 已有   | 侧边栏任务池 + 时间轴排班 |
| 执行 & 记录        | ✅ 已有   | 任务状态、完成标记        |
| 偏差检测 & 预警    | ❌ 缺失   | 无预算 vs 实际对比        |
| 复盘 & 资产沉淀    | ❌ 缺失   | 仅有基础 Insights         |

## 五、高能时段感知（新增）

### 5.1 目标

帮助用户发现 **属于自己的高能量时间段**，从而：
- 把深度工作安排在高能时段
- 把会议、杂务安排在低能时段
- 提升时间 ROI

### 5.2 感知方式（待定）

| 方式         | 说明                                   | 优劣                           |
| ------------ | -------------------------------------- | ------------------------------ |
| **隐式推断** | 从任务完成时长、偏差率、返工率推断效率 | 无需用户额外操作，但数据需积累 |
| **显式反馈** | 任务完成时让用户打分（精力/满意度）    | 数据直接，但增加用户负担       |
| **外部数据** | 对接日历、屏幕时间、健康数据           | 信息丰富，但隐私敏感           |

### 5.3 输出

- **个人效率画像**：时段 × 任务类型 × 成功率/偏差
- **高能时段推荐**：基于历史数据推荐最佳执行时段
- **智能排班建议**：自动把 L4 任务推荐到高能时段

### 6.1 问题

- 用户同时使用 Claude、ChatGPT、Cursor、各种 Agent
- 每个 AI 工具各自推送结果

> 成为用户的 **AI 结果统一调度器**

### 6.3 能力设想

| 能力           | 说明                                      |
| -------------- | ----------------------------------------- |
| **AI Inbox**   | 统一收集各 AI 工具的结果                  |
| **验收窗口**   | 用户设定固定的验收时段（如 11:30、17:30） |
| **深度块保护** | 深度工作时段默认屏蔽非紧急通知            |
| **批量验收**   | 在验收窗口批量处理 AI 结果                |
| **返工统计**   | 记录验收结果，统计返工率                  |

## 七、AI 四层架构：从通才到专家

> **核心理念**：把 LLM 这个"通才毕业生"，通过私有数据、传感器、执行器和反馈闭环，训练成 **你的私人时间教练**。

### 7.1 私有数据层（Private Data）

| 数据类型       | 说明                     | 价值                 |
| -------------- | ------------------------ | -------------------- |
| **目标数据**   | OKR、项目、想法池        | AI 理解"什么重要"    |
| **时间记录**   | 任务、时间块、深度标签   | AI 理解"时间花在哪"  |
| **效能信号**   | 深度工作时长、碎片化指数 | AI 理解"效率如何"    |
| **执行者数据** | 人工/AI协作/AI自动       | AI 理解"谁在执行"    |
| **主观反馈**   | 精力评分、满意度         | AI 理解"感受如何"    |
| **返工记录**   | AI 任务的返工轮次、原因  | AI 理解"AI 协作质量" |

### 7.2 传感器层（Sensors）

| 传感器                   | 触发场景                 | 输出示例                                |
| ------------------------ | ------------------------ | --------------------------------------- |
| **目标偏离检测**         | 用户安排任务时           | "这个任务与本周 O1 目标关联度低"        |
| **时间结构预警**         | 每日规划时               | "今天计划的深度工作时间只有 45min"      |
| **碎片化感知**           | 任务间隔过短时           | "上午有 6 个切换，建议合并"             |
| **能量匹配检测**         | 高难任务被安排在低能时段 | "L4 任务被安排在 16:00，历史效率较低"   |
| **（新增）WIP 过载预警** | 在制 AI 任务过多时       | "当前有 5 个 AI 任务待验收，建议先处理" |

### 7.3 执行器层（Actuators）

| 执行器               | 功能                       | 示例                                    |
| -------------------- | -------------------------- | --------------------------------------- |
| **智能日程规划**     | 基于目标和精力曲线自动排布 | "为你规划今日深度工作块"                |
| **任务自动分类**     | 识别标题自动填充标签       | "已标记为 L3 + ai_assisted"             |
| **时间块推荐**       | 推荐最佳时段               | "建议安排在明早 09:30（你的高效时段）"  |
| **（新增）验收调度** | 把 AI 结果推送到验收窗口   | "3 个 AI 任务已完成，安排在 17:30 验收" |

### 7.4 反馈闭环层（Feedback Loop）

| 反馈信号             | 数据来源          | AI 学习内容      |
| -------------------- | ----------------- | ---------------- |
| **实际 vs 计划差异** | 时间记录          | 学习时长估计偏差 |
| **任务完成率**       | 状态变更          | 学习执行力模式   |
| **手动调整**         | 用户修改 AI 建议  | 学习个人偏好     |
| **（新增）返工率**   | AI 任务验收结果   | 学习 AI 协作质量 |
| **（新增）高能时段** | 效率推断/主观反馈 | 学习最佳执行时段 |

## 八、任务字段定义（更新）

### 8.1 新增字段

| 字段             | 类型   | 说明                                                                      |
| :--------------- | :----- | :------------------------------------------------------------------------ |
| `execution_mode` | enum   | 执行模式：`L0_Manual` / `L1_Copilot` / `L2_Delegated` / `L3_Automated`    |
| `ai_state`       | enum   | AI 状态：`idle` / `running` / `review_needed` / `reworking` / `completed` |
| `ai_tool`        | string | 使用的 AI 工具（如 Claude、ChatGPT）                                      |
| `rework_count`   | int    | 返工轮次                                                                  |
| `cost_stats`     | json   | 成本统计 `{ human_ms, ai_runtime_ms }`                                    |

### 8.2 现有字段（保留）

- `Date`：日期
- `StartTime`：开始时间
- `EndTime`：结束时间
- `DurationH`：时长
- `RawTitle`：标题
- `MainTag`：主类型
- `Depth`：深度等级（L1-L4）
- `R_Tag`：责任归属
- `Battle`：项目/战场
- `OKR_O_ID`：目标 ID
- `OKR_KR_ID`：关键结果 ID
- `Is_OKR`：是否 OKR 相关
- `Signal`：信号/噪音
