# 质量保障机制 (Quality Assurance Mechanism)

> **核心目标**：在敏捷迭代中，确保新功能的开发不破坏已有功能的稳定性。
> **适用范围**：所有前端/后端开发任务。

## 1. 现状回顾与痛点 (Current State)

| 维度           | 当前机制           | 存在问题                            | 典型案例                                 |
| :------------- | :----------------- | :---------------------------------- | :--------------------------------------- |
| **静态检查**   | TypeScript, ESLint | 依赖配置不严，存在 `any` 或导入错误 | `MenuProps` 值被错误导入导致的白屏       |
| **自动化测试** | **无**             | 纯靠人工回归，效率低且易漏测        | 拖拽算法修改导致任务时间计算错误         |
| **代码架构**   | 组件化拆分较为随意 | 存在“上帝组件”，逻辑耦合严重        | `ExecutePage` 耦合了拖拽、渲染、数据逻辑 |
| **开发流程**   | 直接开发           | 缺乏原子化提交与重构隔离            | 修改 A 功能顺手改了 B 功能逻辑导致回归   |

## 2. 防御体系建设 (Three Lines of Defense)

### 第一道防线：架构隔离与静态防御
**原则**：即使代码写得烂，因为有物理/逻辑隔离，也不会炸掉核心功能。

1.  **逻辑抽离 (Logic Extraction)**
    *   **规范**：超过 300 行的组件必须拆分。复杂逻辑（如拖拽、计算）必须抽离为 `hooks` 或 `utils`。
    *   **落地**：
        *   拖拽逻辑 $\rightarrow$ `useDraggableLogic.ts`
        *   时间计算 $\rightarrow$ `timeUtils.ts` (纯函数，易测试)
2.  **类型防御 (Strict Typing)**
    *   **规范**：
        *   禁止使用 `any`。
        *   导入类型必须使用 `import type`。
        *   开启 `tsconfig` 的 `verbatimModuleSyntax` 检查。

### 第二道防线：自动化测试网
**原则**：机器负责验证逻辑正确性，人负责验证体验。

1.  **单元测试 (Unit Tests) - 覆盖核心算法**
    *   **工具**：Vitest
    *   **必测范围**：
        *   `src/utils/*.ts` (时间转换、数据格式化)
        *   `src/stores/*.ts` (状态管理逻辑)
        *   `customCollisionDetection` (拖拽碰撞算法)
    *   **标准**：核心 Utils 覆盖率 100%。

2.  **组件快照 (Snapshot Tests) - 锁定 UI 结构**
    *   **工具**：Vitest + React Testing Library
    *   **范围**：基础组件 (`ScheduleCard`, `TaskModal`)。
    *   **作用**：一旦 UI 结构发生非预期变化（如误删 class），测试立即报错。

### 第三道防线：流程规范 (Process Check)
**原则**：慢即是快，重构与新功能分离。

1.  **原子化提交 (Atomic Commits)**
    *   **规范**：
        *   `feat`: 新功能
        *   `fix`: 修复 Bug
        *   `refactor`: 纯重构（不改变功能行为）
    *   **禁忌**：严禁在一个 commit 中同时包含 `feat` 和 `refactor`。

2.  **功能开关 (Feature Toggles)**
    *   **规范**：对于高风险核心功能（如全新的拖拽引擎），需通过配置项控制。
    *   **代码示例**：
        ```typescript
        const USE_NEW_DRAG_LOGIC = true; // 紧急情况可改为 false 回滚
        ```

## 4. 开发者自查表 (Pre-commit Checklist)

在提交代码前，请按以下问题自问：
1.  **改动范围**：我是否修改了公共组件/工具函数？如果是，是否检查了所有引用处？
2.  **测试验证**：我是否为新写的算法函数添加了单元测试？
3.  **样式隔离**：我的 CSS 修改是否会污染全局？（是否使用了 Scoped CSS 或 Tailwind）
4.  **类型安全**：是否有新增的 `any`？是否有各种 Lint 警告？
