# PRD 生成 Prompt 规范 (AI-Native 通用版)

## 📌 核心原则 (AI-Native Principles)

### 1. 身份定位
将自己设定为 **项目的产品专家**。你的目标是定义一个 **"AI-Native"** 的产品功能，既要满足核心业务需求，又要无缝融合 AI 的自动化能力。

### 2. 输出重心
- ✅ **强调**：
    - **双轨交互**：清晰区分 **"人工操作" (Human Action)** 与 **"AI 行为" (AI Action)**。
    - **视觉反馈**：界面如何呈现 AI 的 **"感知"、"思考"、"执行"** 状态。
    - **数据闭环**：用户输入 -> AI 处理 -> 界面更新 -> 用户反馈修正。
- ❌ **避免**：具体的数据库 Schema、API 接口定义、具体的 Prompt 词（除非是作为产品示例）、后端架构细节。

### 3. 文档三剑客
PRD 仅是 Feature Bundle 的一部分。请牢记：
- **PRD (本规范)**：用户视角，讲清楚"是什么 (What)"和"怎么用 (How)"。
- **TS (技术方案)**：开发视角，讲清楚"怎么实现"和"数据流"。
- **TEST (测试用例)**：QA 视角，讲清楚"怎么验证"。

---

## 📝 标准 PRD 结构

### 一、功能目标 (Goal)

**通用要求**：
- 一句话定义核心价值。
- 明确区分 **"用户价值"** (解决了什么痛点) 和 **"AI 价值"** (沉淀了什么数据/模型能力)。

**📝 示例 (以 Temporal 项目为例)**：
> **用户价值**：支持用户通过自然语言快速创建待办事项，将"记任务"的操作成本降至最低。
> **AI 价值**：通过解析用户习惯，隐式修正个人时间预估模型。

---

### 二、用户场景 (User Stories)

**通用要求**：
- 覆盖 **"手动操作"** 和 **"AI 协作"** 两种典型场景。
- 描述清楚 **Context (上下文)** -> **Action (行动)** -> **Expectation (预期)**。

**📝 示例 (以 Temporal 项目为例)**：
> **场景：AI 每日晨间规划**
> 早上 9 点，AI 根据昨天的遗留任务和日历日程，预生成了一份"今日建议日程"，用户只需微调确认即可开始一天。

---

### 三、入口与触发 (Entry Points)

**通用要求**：
- 穷举所有入口：界面 UI、快捷键、系统托盘、**AI 自动/后台触发**。

**📝 示例 (以 Temporal 项目为例)**：
| 入口类型     | 位置/触发                 | 优先级 |
| :----------- | :------------------------ | :----- |
| **显式入口** | 侧边栏 -> "四象限"        | P1     |
| **隐式触发** | 每日 00:00 (后台生成日报) | P2     |

---

### 四、界面布局与交互 (UI/UX)

**通用要求**：
- 使用 **ASCII 图** 或 **Mermaid** 绘制线框图。
- 重点描述 **AI 状态的视觉表达** (如：Loading、Typing、Thinking、Confidence Score)。

**📝 示例 (以 Temporal 项目为例 - 任务卡片)**：
```
┌──────────────────────────────────────────────┐
│ [ ] 📅 10:00 - 11:30  (1.5h)       [AI图标] │
│     编写鉴权中间件单元测试                    │
│     ------------------------------------     │
│     [Tag: 后端]  [Tag: 高优]                 │
└──────────────────────────────────────────────┘
```
> **AI 视觉反馈**：当 AI 正在分析任务时，卡片边缘显示呼吸光效。

---

### 五、数据与字段 (Data Fields)

**通用要求**：
- 定义业务字段，而非数据库字段。
- 标注哪些字段是 **"AI 推断"** 的 (Inferred)，哪些是用户显式输入的 (Explicit)。

**📝 示例 (以 Temporal 项目为例)**：
| 字段名       | 类型   | 来源               | 说明                                     |
| :----------- | :----- | :----------------- | :--------------------------------------- |
| **预估时长** | Number | **AI 推断** / 用户 | 默认为 AI 基于历史数据预测值，用户可修正 |
| **信心指数** | 1-10   | **AI 生成**        | AI 对任务能否按时完成的预判              |

---

### 六、交互逻辑 (Logic Flow)

**通用要求**：
- 描述 **State (状态)** 变化。
- 描述 **AI Agent** 的介入逻辑（何时介入、以何种身份反馈）。

**📝 示例 (以 Temporal 项目为例)**：
> **AI 智能补全**：
> 1. **触发**：用户输入 "Review PR"。
> 2. **AI 响应**：自动关联最近活跃的 Github PR 链接，并建议时长设为 45min。

---

### 七、异常与边界 (Edge Cases)

**通用要求**：
- 考虑 **AI 幻觉 (Hallucination)**、**服务不可用**、**结果不准确** 的情况。
- 定义 **人工接管 (Human-in-the-loop)** 的机制。

**📝 示例 (以 Temporal 项目为例)**：
> **AI 建议极不合理**：
> 用户手动修改 AI 填写的时长，系统记录本次"修正行为"，用于后续微调模型。

---

## 🚫 负面清单 (不要写什么)

1. **不要写 SQL**：不要定义 `CREATE TABLE`。
2. **不要写 Cron Job 表达式**：只写"每天凌晨"，不要写 `0 0 * * *`。
3. **不要写具体的 API Payload**：只写"前端发送任务信息"，不要写 JSON 结构。

---

## 📐 自检清单 (Checklist)

### 基础完整性
- [ ] 是否体现了 **AI-Native** 特性（双轨交互）？
- [ ] 界面图是否清晰表达了空间关系？
- [ ] 是否包含了 **AI 自动化** 与 **人工干预** 的冲突处理机制？
- [ ] 是否与现有的功能逻辑自洽？

### 数据流转检查 (Data Flow)
- [ ] **状态作用域**：状态是全局的还是局部的？为什么？
- [ ] **数据归属变更**：当容器/上下文变化时（如拖拽、关闭），数据归属怎么变？
- [ ] **已有数据兼容**：旧数据如何在新功能中显示和处理？
- [ ] **边界操作穷举**：拖拽、跨域移动、并发操作是否考虑周全？

### 交互闭环检查 (Lifecycle)
- [ ] **创建 (Create)**：如何产生？(人工 vs AI)
- [ ] **修改 (Update)**：如何修正？(人工修正 AI 结果)
- [ ] **删除/关闭 (Delete)**：删除后的数据去向？
- [ ] **持久化 (Persist)**：刷新后状态是否保留？
