# PRD: 分屏时间轴 (Split-Pane Timeline) - Phase 1.5

> **前置依赖**：本功能依赖 [03_项目管理基础](../03_项目管理基础/PRD_需求文档.md)，需要 Project 实体已建立。

## 1. 功能目标

让用户能够将时间轴区域拆分为多个并行的 **泳道 (Swimlanes)**，实现多项目/多上下文的并排查看与管理。

- **用户价值**：解决"多项目并行焦虑"，提供类似 IDE 多标签页的分窗体验，让用户在一个视图内完成跨项目调度。
- **AI 价值**：为未来 AI 按项目上下文进行智能规划提供数据基础 (`projectId` 关联)。

---

## 2. 用户场景

### 场景 A：多项目并行查看 (Multi-Project View)
> 产品经理同时负责"白板功能开发"和"日常事务"两条线，希望一边看项目任务，一边安排常规会议。
> **诉求**：点击"添加分屏"，时间轴分成两列，左列看白板任务，右列看日常杂事。

### 场景 B：调整列宽 (Resize Lanes)
> 用户发现"白板项目"这轨任务较多，卡片挤在一起看不清。
> **操作**：鼠标悬停在两列之间的分割线上，左右拖拽，扩大白板列的宽度。

### 场景 C：关闭多余列 (Close Lane)
> 开完会后，用户不再需要分屏，想回到单列模式。
> **操作**：点击第二列 Header 上的"关闭"按钮，界面恢复单列。

### 场景 D：跨泳道移动任务 (Cross-Lane Drag)
> 用户发现一个任务本来放在"日常杂事"里，其实应该属于"白板项目"。
> **操作**：将任务卡片从左侧泳道拖拽到右侧泳道，任务自动关联新的项目上下文。

### 场景 E：从待办池拖入指定泳道 (Sidebar to Lane)
> 左侧待办池有一个未排班的任务，用户想把它排到"白板项目"的泳道里。
> **操作**：从左侧拖拽任务卡片，放到右侧"白板项目"泳道的某个时间点。

### 场景 F：自定义泳道名称 (Rename Lane)
> 用户觉得默认的 "Project Lane" 名字不够直观，想改成 "白板项目"。
> **操作**：双击泳道 Header 的标题，进入编辑模式，输入新名字后回车确认。

### 4.1 整体布局

```
┌─────────────────────────────────────────────────────────────────────┐
│ [Header: 日期选择器]                           [统计信息]            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────┐ │ ┌─────────────────────────┐         │
│  │ Focus Track         [+]│ │ │ 白板项目           [×]│         │
│  ├─────────────────────────┤ │ ├─────────────────────────┤         │
│  │ 09:00 [ Task A ]       │◄┼─┤ 09:00 [ Task X ]       │         │
│  │ 10:00 [ Task B ]       │ │ │ 10:00    ↑拖拽到此     │         │
│  │ 11:00                  │ │ │ 11:00 [ Task Y ]       │         │
│  │ ...                    │ │ │ ...                    │         │
│  └─────────────────────────┘ │ └─────────────────────────┘         │
│                             ↕                                       │
│                        [分割线拖拽]                                  │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 泳道 Header 规格

| 元素         | 说明                             |
| :----------- | :------------------------------- |
| **标题**     | 显示泳道名称，**双击可编辑**     |
| **添加按钮** | 仅在第一列显示，点击后新增一列   |
| **关闭按钮** | 仅在非第一列显示，点击后关闭该列 |

### 4.3 分割线交互

| 状态       | 表现                                  |
| :--------- | :------------------------------------ |
| **默认**   | 透明，宽度 4px，但可点击区域 8px      |
| **Hover**  | 背景变为淡蓝色，光标变为 `col-resize` |
| **拖拽中** | 背景变为深蓝色，实时更新左右两列宽度  |

| 字段名        | 类型              | 来源      | 说明                                          |
| :------------ | :---------------- | :-------- | :-------------------------------------------- |
| **lanes**     | Array<LaneConfig> | 系统/用户 | 泳道配置，包含 id、title、projectId、width    |

| **width**     | Number (%)        | 用户拖拽  | 泳道宽度百分比，总和为 100%                   |
| **projectId** | String?           | 用户选择  | 关联项目 ID，用于筛选任务（可选）             |

## 6. 交互逻辑

### 6.1 添加泳道
1. **动作**：用户点击第一列 Header 的 `+` 按钮。
2. **系统响应**：
    - 新增一列泳道，默认标题为 "Project Lane"。
    - 所有泳道宽度自动重新分配（均分）。
    - 新泳道默认不绑定项目，显示所有任务。

### 6.2 调整列宽
1. **动作**：用户拖拽分割线。
2. **系统响应**：
    - 实时更新左右两列的 `width` 值。
    - 任务卡片宽度自适应。
    - **最小宽度限制**：每列最小 10%，防止拖到看不见。

### 6.3 关闭泳道
1. **动作**：用户点击非首列的 `×` 按钮。
2. **系统响应**：
    - 移除该泳道配置。
    - 剩余泳道宽度自动重新分配（均分）。
    - **数据处理**：泳道关闭 **不影响任务数据**，任务的 `projectId` 保持不变。被关闭泳道中的任务会在主轨道 (Focus Track) 中继续显示（因为主轨道显示所有任务）。
    - **至少保留一列**：首列无法关闭。

### 6.4 重命名泳道
1. **动作**：用户双击泳道 Header 的标题区域。
2. **系统响应**：
    - 标题变为可编辑的 Input 框，显示当前名称。
    - 用户输入新名称后，按 Enter 或点击外部区域确认修改。
    - 按 Esc 取消修改。

### 6.5 跨泳道拖拽任务
1. **动作**：用户将一个泳道内的任务卡片拖拽到另一个泳道。
2. **系统响应**：
    - **更新任务时间**：根据落点位置计算新的 `planned_start`。
    - **更新任务关联** (可选)：如果目标泳道绑定了特定 `projectId`，任务的 `projectId` 也随之更新。如果目标泳道未绑定项目（如 Focus Track），则保持任务原有 `projectId` 不变。

### 6.6 从待办池拖入泳道
1. **动作**：用户从左侧 Sidebar 的待办池拖拽未排班任务到某个泳道。
2. **系统响应**：
    - 为任务创建 `schedule`（排班信息）。
    - 如果目标泳道绑定了 `projectId`，同时更新任务的 `projectId`。

### 6.7 任务显示规则
1. **主轨道 (Focus Track)**：`projectId = undefined`，显示 **所有任务**（包括有/无 projectId 的任务）。
2. **项目轨道 (Project Lane)**：`projectId = 'xxx'`，**仅显示** 该项目关联的任务。

## 7. 泳道作用域设计

### 7.1 核心决策：泳道配置是 **全局** 的，而非按日期独立

| 方案                | 描述                         | 优点                             | 缺点                     |
| :------------------ | :--------------------------- | :------------------------------- | :----------------------- |
| **全局泳道 (采用)** | 所有日期共享同一套泳道配置   | 简单直观，切换日期不需要重新配置 | 无法针对特定日期定制布局 |
| 按日期泳道          | 每个日期可以有独立的泳道配置 | 灵活                             | 复杂，用户心智负担重     |

**结论**：采用 **全局泳道**，因为：
1. 用户的项目通常是跨天的，每天重新配置太繁琐。
2. MVP 阶段优先简单可用，复杂需求留待未来迭代。

### 7.2 切换日期时的行为
- 泳道数量、宽度、名称 **保持不变**。
- 各泳道内显示的任务 **根据选中日期动态过滤**。

## 9. 异常与边界

| 异常场景         | 表现                            | 托底方案                              |
| :--------------- | :------------------------------ | :------------------------------------ |
| **屏幕过窄**     | 列宽低于 200px 时无法再添加泳道 | 提示 "屏幕宽度不足，无法添加更多分屏" |
| **拖拽到极限**   | 任一列宽度接近 10% 时           | 阻止继续拖拽，分割线不再移动          |
| **关闭唯一泳道** | 首列无 `×` 按钮                 | 从 UI 上禁止此操作                    |
| **泳道名称为空** | 用户清空输入框后确认            | 恢复为默认名称 "Project Lane"         |
