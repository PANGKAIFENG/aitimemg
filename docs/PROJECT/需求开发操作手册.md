# 需求开发操作手册

## 这份手册解决什么问题

这份手册是给产品经理和研发一起用的。

它回答的是：

- 一个新需求来了，先做什么
- 产品、研发、Reviewer 各自要做什么
- 代码什么时候可以合并
- 什么时候可以发布
- 发布后要怎么验收

当前项目统一遵循这条主线：

`需求澄清 -> 分支开发 -> 提交 PR -> Review -> CI 通过 -> 合并 main -> 从 main 发布`

---

## 一、先判断这次需求属于哪一类

### 1. 小改动 / 明确需求

适合直接进入开发的典型情况：

- 文案调整
- 一个清晰的小交互优化
- 已经明确范围的 bug 修复
- 不涉及多页面、多角色、多状态联动

这种需求可以直接进入“分支开发”。

### 2. 中等功能

适合先补最小需求说明的情况：

- 新增一个页面模块
- 新增一个完整交互
- 涉及前后端配合
- 有状态变化或边界处理

这种需求建议先补一份简版说明，至少写清：

- 背景
- 目标
- 范围
- 成功标准
- 风险点

### 3. 复杂功能

符合下面任一条，就不要直接开写代码：

- 涉及多步骤流程
- 涉及 Agent / AI / Context / Workflow
- 涉及多页面联动
- 涉及多人协作角色
- 需求本身还在脑爆阶段

这种需求应先补：

- PRD
- 必要时的流程图 / ASCII 草图
- handoff

---

## 二、你作为产品经理，需求来了先怎么做

### 1. 先把需求说清楚

至少要说明这 6 件事：

1. 为什么做这个需求
2. 给谁用
3. 用户当前的问题是什么
4. 你希望改完后用户能做到什么
5. 范围边界是什么
6. 怎么判断这次改动算成功

### 2. 判断要不要先写文档

- 如果研发拿到后能直接明确范围，可以不先写正式 PRD
- 如果还会继续追问边界和状态，就先补文档

当前项目文档位置：

- PRD：`docs/PRD/`
- 项目上下文：`docs/PROJECT/`
- 决策记录：`docs/DECISIONS/`
- handoff：`docs/HANDOFF/`

---

## 三、研发收到需求后怎么做

### 第一步：先看当前工位状态，再开干净分支工位

不要直接在 `main` 上开发。

如果当前仓库里还有旧改动，优先用 worktree 隔离，不要在脏工作区里直接切新分支。

分支命名规则：

- 新功能：`feature/<topic>`
- 修 bug：`fix/<topic>`
- 流程/文档/脚本：`chore/<topic>`

示例：

```bash
cd /Users/linctex/Desktop/vibe/时间管理
bash scripts/worktree-status.sh
bash scripts/worktree-new-feature.sh today-plan-entry
cd ~/.config/superpowers/worktrees/时间管理/feature-today-plan-entry
git status
```

### 第二步：在分支上开发

开发过程中只在自己的分支上改，不把半成品合到 `main`。

### 第三步：本地自测

#### 前端验证

```bash
cd temporal
npm run build
```

#### 后端语法检查

```bash
cd temporal/aliyun-fc/api
node -c index.js
if [ -d src ]; then find src -name '*.js' -print0 | xargs -0 -n1 node -c; fi
```

### 第四步：提交 commit 并 push 分支

```bash
git add <具体文件>
git commit -m "feat: add today plan entry flow"
git push -u origin feature/today-plan-entry
```

## 四、PR 是什么，什么时候提

PR 可以理解成：

> “我这次改动开发完了，请帮我 review，确认是否能进正式版本。”

PR 里至少要写清：

- 这次改了什么
- 为什么改
- 前端 / 后端 / 文档影响范围
- 风险和回滚方式
- 怎么验证
- 是否需要发前端 / 后端

模板位置：

- `.github/pull_request_template.md`

---

## 五、CI 是什么

CI 可以理解成“系统自动做的基础验收”。

当前项目最小 CI 只卡两件事：

- `Frontend Build`
- `Backend Syntax`

所以规则是：

- CI 没过，不合并

---

## 六、什么时候可以合并到 `main`

满足下面几个条件后，PR 才应该 merge：

1. 需求方向确认没跑偏
2. 至少一个 reviewer approve
3. CI 通过
4. 风险已知
5. 发布影响清楚

---

## 七、怎么发布

### 前端发布

```bash
cd temporal
bash scripts/release-frontend-from-main.sh
```

### 后端发布准备

```bash
cd temporal
bash scripts/release-backend-prepare-from-main.sh
```

当前统一规则是：

> 只允许从干净的 `main` 发布。

## 八、你以后最常用的一套问法

1. 这个需求对应哪个分支？
2. PR 链接发我。
3. CI 过了吗？
4. Reviewer 过了吗？
5. 这次影响前端、后端，还是两边？
6. 合并到 `main` 了吗？
7. 上线了吗？对应哪个 commit？
