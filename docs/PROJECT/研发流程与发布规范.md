# 研发流程与发布规范

## 1. 当前真实流程

基于当前仓库和脚本，Temporal 现在的研发流程实际上是：

1. 直接在本地 `main` 开发
2. 本地验证，主要依赖手工跑 `npm run build`
3. 前端直接执行 `temporal/package.json` 里的 `npm run deploy`
4. `deploy` 本质是运行 [deploy-oss.cjs](/Users/linctex/Desktop/vibe/时间管理/temporal/scripts/deploy-oss.cjs:1)，把前端构建产物直接上传到阿里云 OSS
5. 后端不是同一条自动流程，当前是：
   - 先用 [build-api.sh](/Users/linctex/Desktop/vibe/时间管理/temporal/scripts/build-api.sh:1) 打 zip
   - 再人工去阿里云 FC 控制台上传

当前仓库状态也说明，这套流程还没有真正规范住：

- 远端只有 `main`，没有稳定使用中的 `develop`
- 当前没有 `.github/workflows/`，也就是没有 GitHub Actions CI
- 当前没有 PR 模板
- 当前没有 CODEOWNERS
- 现有文档里虽然写了“禁止直接 push main”，但仓库里没有真正执行这条规则的闸门

## 2. 当前问题

现在这套流程最大的问题不是“没有文档”，而是“文档和真实执行脱节”。

核心问题有 5 个：

1. `main` 既是开发分支，也是生产分支  
这会导致未 review 的代码直接进入线上。

2. 发布动作和代码合并动作没有解耦  
现在是谁本地有权限，谁就能直接 `npm run deploy`。

3. 前后端不是一条统一发布链  
前端是 OSS 直传，后端是 FC 控制台人工上传，可追溯性很差。

4. 没有自动 review 闸门  
没有 PR 模板、没有 reviewer 责任人、没有 CI 状态检查。

5. 质量门槛不清晰  
当前 `build` 能过，但 `lint` 还存在历史问题，不能直接作为强制门槛。

## 3. 目标流程

现阶段最适合 Temporal 的目标流程，不是一下子上很重的 DevOps，而是先收敛成这条最小闭环：

`分支开发 -> 提交 PR -> Review -> 合并 main -> 从 main 发布`

具体来说：

1. 从 `main` 拉 `feature/*` 或 `fix/*` 分支
2. 在分支上开发和自测
3. 推送分支并发起 PR 到 `main`
4. 通过最小 CI
5. 至少 1 个 reviewer 通过
6. 合并到 `main`
7. 只允许从 `main` 发布到线上

这条流程的重点不是形式，而是把两件事分开：

- “代码写完了”
- “代码可以上线了”

## 4. 现阶段建议的标准

### 4.1 分支规范

- `main`
  - 生产分支
  - 只接受 PR 合并
  - 不允许直接 push
- `feature/<topic>`
  - 新功能开发
- `fix/<topic>`
  - Bug 修复
- `chore/<topic>`
  - 脚本、流程、文档、基础设施调整

### 4.2 PR 规范

每个 PR 至少回答清楚 5 件事：

1. 这次改什么
2. 为什么改
3. 风险点是什么
4. 怎么验证
5. 是否需要发布前端 / 后端

### 4.3 Review 规范

第一阶段先要求：

- 至少 `1` 个 review approval
- 作者不能自己 approve 自己
- 有风险改动必须写回滚方式

### 4.4 CI 规范

当前仓库适合先上“最小可执行 CI”，不要一开始就拿 `lint` 卡住全团队。

第一阶段强制检查：

- 前端 `npm run build`
- 后端 `node -c` 语法检查

第一阶段暂不强制：

- 前端 `npm run lint`

原因很明确：当前 lint 里还有较多历史存量问题，先拿它做硬门槛会让流程形式化但无法落地。

### 4.5 发布规范

现阶段建议这样执行：

1. 功能分支合并到 `main`
2. 在 `main` 上确认本次发布范围
3. 前端从 `main` 发布
4. 如涉及后端，再从 `main` 构建并发布后端
5. 发布后做线上验收
6. 记录发布结果

也就是说：

- 不允许“分支代码没合并就直接发线上”
- 不允许“本地脏工作区直接当生产真相源”

## 5. 这次已经落到仓库里的约束

这轮会把下面三类约束加进仓库：

1. `PR Template`
2. `CODEOWNERS`
3. `GitHub Actions CI`

这三类约束都属于“仓库级约束”，不靠口头执行。

## 6. GitHub 侧还需要手工打开的设置

仓库内文件只能提供约束基础，真正让流程生效，还需要在 GitHub 仓库设置里打开：

### `main` 分支保护建议

- Require a pull request before merging
- Require approvals: `1`
- Dismiss stale approvals when new commits are pushed
- Require status checks to pass before merging
- Restrict who can push to matching branches
- Disable force push

### 建议启用的状态检查

- `Frontend Build`
- `Backend Syntax`

## 7. 前后端发布的建议拆分

当前最大的流程风险，在于前后端发布方式不一致。

### 前端

当前已经是脚本化上传 OSS，适合优先规范成：

- 合并 `main`
- 由发布人从 `main` 执行发布

### 后端

当前还是：

- 本地打包
- 人工去 FC 控制台上传

这一步仍然偏手工，建议下一阶段再补：

- 后端发布脚本
- 或 GitHub Actions 的手动发布工作流

但在补完之前，至少要做到：

- 后端包只能从 `main` 构建
- 后端上线必须记录对应 commit SHA

## 8. 建议的最小执行口径

从现在开始，建议团队统一按下面这条口径执行：

1. 不再直接在 `main` 上开发新功能
2. 所有功能先走分支
3. 所有合并先走 PR
4. 所有上线只从 `main` 发
5. 前端和后端上线都要能追溯到一个明确 commit

## 9. 下一步建议

如果要继续把流程做完整，优先级建议是：

1. 先打开 GitHub `main` 分支保护
2. 开始强制走 PR + review
3. 保持 CI 只卡 `build / syntax`
4. 再补一个“发布记录”文档或 release note 机制
5. 最后再把后端发布自动化

这样推进最稳，不会一开始把流程搞得很重，但又能马上挡住“直接改 main 直接发线上”的问题。
